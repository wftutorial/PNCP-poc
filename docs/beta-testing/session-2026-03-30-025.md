## ISSUE-049: Busca Saude timeout na primeira tentativa — crash frontend "total_oportunidades"

**Persona:** Marcus (Performance) + Priya (Edge Cases)
**Severidade:** P2
**Area:** Buscar — error handling
**Data:** 2026-03-30 09:05
**Status:** ABERTO

### Classificacao de Severidade
P2 — O timeout aconteceu durante restart do backend (transiente), mas o crash do frontend (TypeError: Cannot read properties of undefined reading 'total_oportunidades') deveria ser tratado gracefully. O retry manual funcionou.

### Passos para Reproduzir
1. Navegar para https://smartlic.tech/buscar
2. Selecionar setor "Saude", UF "SP", status "Abertas"
3. Clicar "Buscar Saude"
4. Aguardar ~30s — progress bar ativa, SP "Aguardando..."
5. Frontend exibe "Erro na busca" com mensagem tecnica

### Comportamento Observado
- POST /api/buscar falhou com `net::ERR_ABORTED` (2 tentativas)
- SSE /api/buscar-progress conectou OK (200) mas nao recebeu dados
- Warning: "[CRIT-070] Client timeout triggered"
- Error: "TypeError: Cannot read properties of undefined (reading 'total_oportunidades')"
- Tela de erro mostra mensagem tecnica crua ao usuario

### Comportamento Esperado
1. Frontend deve tratar gracefully o cenario de timeout — mostrar mensagem amigavel sem expor TypeError
2. Botao "Tentar novamente" deve preservar o setor selecionado (reverteu para Vestuario)
3. Se backend esta reiniciando, o banner "Servidor reiniciando..." deveria aparecer ANTES da busca, nao durante

### Screenshot
`docs/beta-testing/screenshots/session-025/saude-sp-error.png`

### Notas Tecnicas
- Backend estava reiniciando (deploy do commit `d59a7d3e` provavelmente em andamento)
- Banner "Servidor reiniciando..." apareceu durante a 2a tentativa, indicando health check detectou indisponibilidade
- O campo `total_oportunidades` esta sendo acessado sem optional chaining no `page.tsx`
- Console errors: `page-d945aa12b6a56453.js:5:11259` — buscar/page.tsx compilado

### Fix Sugerido
1. Adicionar optional chaining: `resultado?.resumo?.total_oportunidades` ou guardar com `if (!resultado?.resumo) return <ErrorState />`
2. "Tentar novamente" deve preservar o setor selecionado (usar state do formulario, nao default)
3. Considerar mostrar banner de backend indisponivel proativamente quando health check falha, antes de permitir busca
