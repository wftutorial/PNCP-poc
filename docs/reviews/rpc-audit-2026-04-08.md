Esta auditoria cobre o universo completo de RPCs remanescentes.

## Findings e Correções

### FINDING-01: `search_datalake` — bypass de quota via acesso direto

**Severidade:** MÉDIA
**Risco:** Usuário autenticado pode chamar `search_datalake` diretamente via PostgREST,
contornando o sistema de quotas do backend (que é aplicado em `check_and_increment_quota`).
A função não expõe PII (retorna dados de licitações públicas), mas viola o enforcement de quotas.

**Antes:** `GRANT EXECUTE TO authenticated, service_role`
**Depois:** `GRANT EXECUTE TO service_role` apenas

**Migration:** `20260408220000_debt03_rpc_security_audit.sql`

### FINDING-02: `increment_share_view` — GRANT implícito via PUBLIC default

**Severidade:** BAIXA
**Risco:** PostgreSQL concede EXECUTE a PUBLIC por padrão em novas funções.
A intenção era permitir acesso anônimo (páginas de análise compartilhada são públicas),
mas o GRANT não era explícito, deixando a intenção opaca.

**Antes:** Sem GRANT explícito (comportamento via padrão PostgreSQL)
**Depois:** `GRANT EXECUTE TO anon, authenticated, service_role` (intencional e documentado)

**Migration:** `20260408220000_debt03_rpc_security_audit.sql`

## AC4: Escopo de TD-005 (Per-User Supabase Tokens)

### Contexto Atual

O backend usa `service_role` para todas as operações Supabase via `get_supabase()`.
O `get_user_supabase(access_token)` existe mas é usado apenas em endpoints específicos
(OAuth, sessões de usuário).

### O que TD-005 Resolveria

1. **Auditoria granular:** Logs de operações identificados por usuário em vez de service_role
2. **Defense-in-depth:** RLS aplicada no banco mesmo que o backend tenha bugs de autorização
3. **Conformidade:** Redução de superfície de ataque do service_role

### Análise de Custo-Benefício

**Benefícios:**
- Trilha de auditoria mais rica (quem fez o quê, no banco)
- Falha de segurança no backend não implica acesso irrestrito

**Custos:**
- Refatoração significativa: ~60 funções em routes/ precisam receber o token do usuário
- Latência: token propagation adiciona overhead de autenticação por request
- Complexity: service_role ainda necessário para operações cross-user (ingestion, cron jobs, webhooks Stripe)
- Testes: toda a suite de testes precisa ser adaptada

### Recomendação

**Escopo TD-005:** Implementação de user-scoped tokens em rotas críticas de leitura de dados:
- `GET /me`, `GET /sessions`, `GET /pipeline`, `GET /analytics/*`
- Manter service_role para: webhooks Stripe, ingestion, operações admin, cron jobs

**Prioridade:** P3 — pós-v1.0 (revenue target atingido)
**Estimativa:** 16h de desenvolvimento + 8h de testes
**Pré-requisito:** Refatorar `supabase_client.py` para suportar dual-mode por request context

### Gate de Implementação TD-005

Antes de implementar, verificar:
- [ ] Todas as rotas user-facing passam `access_token` para `get_user_supabase()`
- [ ] Nenhuma rota user-facing usa `get_supabase()` para dados de usuário
- [ ] Cron jobs e webhooks continuam usando `get_supabase()` (service_role)
- [ ] Testes de integração validam RLS enforcement com user token
